# Full-Stack Quality

Use this skill to keep a project simple, consistent, reusable, secure, observable, tested, and maintainable across database, backend, frontend, HTTP API, GraphQL API, delivery/configuration, and shared utility/library work.

## Operating Rules

1. Inspect the existing project conventions before recommending changes.
2. Prefer the smallest change that improves correctness, clarity, or maintainability.
3. Do not introduce a new framework, package, pattern, abstraction, or service unless it removes real complexity.
4. Preserve the user's chosen stack. The skill is language-agnostic; apply the principles to whatever database, backend, and frontend are present.
5. Separate facts from recommendations. Findings must include evidence: path, line or symbol when available, and why it matters.
6. Treat security, data integrity, migrations, auth, and destructive database changes as high-risk. Ask before irreversible changes.
7. If the user asks for code, produce idiomatic code in the project's existing style and include focused validation.
8. Review/audit mode is read-only. Do not edit project code, schemas, configs, generated files, or docs unless the user explicitly asks for implementation or document generation.

## Output Formats

Use the format that matches the task:

- Human review or explanation: Markdown.
- Structured audit findings: JSON using `references/quality-checklist.json` and `templates/review-findings.md` as the human-facing wrapper.
- Architecture decisions: Markdown ADR using `templates/adr.md`.
- Implementation plans: Markdown using `templates/implementation-plan.md`.
- Machine-readable checklist/config generated by this skill: JSON by default.
- YAML: only when the existing project already uses YAML for that artifact, when integrating with a tool that requires YAML, or when the user asks for YAML.

For narrow or small audits, use this two-layer output:

1. `findings.json`: machine-readable findings following `quality-checklist.json`.
2. `review.md`: concise human report grouped by severity and area.

For broad or parallel audits, create per-area JSON files by default:

```text
review.md
findings/
├── database.findings.json
├── backend.findings.json
├── frontend.findings.json
├── http-api.findings.json
├── graphql.findings.json
├── security.findings.json
├── utilities.findings.json
├── testing.findings.json
├── performance.findings.json
├── observability.findings.json
└── delivery.findings.json
```

Only create files for areas that were actually audited. Do not combine broad parallel-audit findings into one large `findings.json` unless the user asks for a single file.

## Review Workflow

1. Classify the task: database, backend, frontend, mobile, HTTP API, GraphQL API, security, privacy, accessibility, i18n, analytics, background jobs, AI/LLM safety, infrastructure, documentation, multi-tenancy, payments/billing, notifications, data import/export, API compatibility, developer experience, testing, performance, observability, delivery/configuration, utilities/libraries, or cross-stack.
2. Load the relevant reference files, auditor file, and `references/quality-checklist.json`.
3. Inventory the current implementation:
   - database: schema source, migrations, models, relations, indexes, constraints;
   - backend: entry points, layers/modules, routes/controllers, services, data access, tests;
   - frontend: pages/routes, components, state, styling, design tokens, forms, accessibility;
   - HTTP: resources, methods, status codes, payloads, auth, validation, errors;
   - GraphQL: schema, resolvers, operations, dataloaders, authz, complexity limits, generated types;
   - security: secrets, authn/authz, dependency risk, injection risk, validation, data exposure;
   - testing: critical paths, test pyramid, brittle tests, missing regression coverage;
   - performance: query patterns, rendering cost, payload size, caching, async work;
   - observability: logs, metrics, traces, alerting, error handling, correlation IDs;
   - delivery/configuration: env config, CI checks, migrations, deploy safety;
   - mobile: platform conventions, state, offline behavior, native permissions, releases;
   - accessibility: keyboard/screen-reader behavior, semantics, contrast, focus, labels;
   - i18n: string extraction, locale formatting, RTL, translation safety;
   - privacy: PII collection, retention, consent, deletion/export, data minimization;
   - analytics: event naming, ownership, duplication, PII leakage, funnel coverage;
   - background jobs: retries, idempotency, dead letters, scheduling, observability;
   - AI/LLM safety: prompt boundaries, tool permissions, evals, data leakage, fallback behavior;
   - infrastructure: IaC, IAM, network exposure, secrets, environment parity;
   - documentation: README/API/onboarding accuracy, stale docs, generated-doc drift;
   - multi-tenancy: tenant scoping, cross-tenant leaks, ownership checks, tenant-aware uniqueness;
   - payments/billing: idempotency, webhook verification, reconciliation, invoices, refunds;
   - notifications: email/SMS/push templates, retries, unsubscribe, delivery tracking;
   - data import/export: parsing, validation, partial failure, idempotency, privacy;
   - API compatibility: versioning, breaking changes, deprecation, client compatibility;
   - developer experience: setup, scripts, seed data, local env, debugging, onboarding;
   - utilities/libraries: shared helpers, duplicate functions, dead code, dependency overlap.
4. Report findings by severity: critical, high, medium, low.
5. Recommend one practical path. Include trade-offs only when there are genuinely competing good options.
6. When implementing, keep edits scoped and validate with the repo's existing tests, linters, type checks, or manual checks.
7. For a formal review, format findings with `templates/review-findings.md`.
8. For an architecture choice, use `templates/adr.md`.
9. For implementation work, use `templates/implementation-plan.md` before editing when the change is broad.

## Project Context Workflow

Use this optional workflow when the user asks to create project docs, guide future agents, track quality over time, decide a target architecture, verify drift, or create a CI quality gate.

1. Audit repo: run the relevant auditors, preferably in parallel for broad audits.
2. Decide target architecture: propose 2-3 realistic target structures, explain trade-offs, and record accepted decisions as ADRs.
3. Generate project docs: create concise docs that guide future agents and developers.
4. Record long-term project memory: decisions, known legacy risks, migration backlog, current sprint/context, and audit history.
5. Compute health score/trend: summarize quality risk consistently across runs.
6. Verify drift: check whether generated docs still match the code.
7. CI quality gate: for PR/diff checks, fail or warn when changes violate accepted architecture, security, testing, or documentation rules.

Read `references/project-context.md` before running this workflow. Use these templates as needed:

- `templates/project-docs-index.md`
- `templates/audit-inventory.json`
- `templates/drift-report.md`
- `templates/migration-backlog.md`
- `templates/ci-quality-gate.md`

## Parallel Auditor Workflow

Use this workflow for broad audits that cover multiple areas.

1. If the environment supports subagents, run one subagent per relevant audit area in parallel.
2. Each auditor subagent is read-only.
3. Give each subagent only the context it needs:
   - `auditors/_common.md`
   - its own area file from `auditors/`
   - the relevant file from `references/`
   - `references/quality-checklist.json`
   - `templates/review-findings.md`
4. For whole-project audits, do not run a single general auditor when subagents are available. Fan out across all relevant areas: database, backend, frontend, mobile, HTTP API, GraphQL, security, privacy, accessibility, i18n, analytics, background jobs, AI/LLM safety, infrastructure, documentation, multi-tenancy, payments/billing, notifications, data import/export, API compatibility, developer experience, utilities, testing, performance, observability, and delivery/configuration.
5. For narrow reviews, run only the matching auditor or auditors. For example, a frontend-only architecture review should use the frontend auditor, not every auditor.
6. Skip auditors for systems the project does not use. For example, skip GraphQL if there is no GraphQL schema, resolver, client operation, or dependency.
7. If subagents are unavailable, run the same auditors sequentially in the main context.
8. Write each auditor's structured output to `findings/<area>.findings.json` when persisting a broad audit.
9. Merge a concise human summary into `review.md` after all auditors finish.
10. Deduplicate overlapping findings by keeping the clearest evidence and preserving all affected areas.
11. Produce Markdown for the human report and JSON for structured findings.

## Design Defaults

- Database: normalized by default; denormalize only for measured read performance or clear product requirements.
- Naming: consistent, readable, and domain-driven. Prefer one convention per layer.
- Backend: clear module boundaries, thin transport handlers, business logic in services/use cases, persistence behind repositories or data-access modules.
- Frontend: shared design tokens, reusable components, predictable state ownership, accessible controls, responsive layouts.
- HTTP: resource-oriented URLs, correct methods, explicit status codes, typed validation, consistent error envelopes, idempotency where needed.
- GraphQL: schema should model domain capabilities clearly, resolvers stay thin, authorization is server-side, N+1 is prevented, and operations are typed/testable.
- Security: validate at boundaries, authorize server-side, keep secrets out of code, avoid logging sensitive data, use maintained dependencies.
- Testing: cover behavior and risk, not implementation noise; add regression tests for bugs.
- Performance: measure first when possible; fix obvious N+1, unbounded lists, large payloads, and expensive renders early.
- Observability: logs should explain what happened without leaking sensitive data; important flows need metrics and traceability.
- Delivery: config should be explicit per environment; CI should catch formatting, typing, tests, security checks, and migration issues when available.
- Mobile: respect platform patterns, permissions, offline/network transitions, release safety, and device constraints.
- Accessibility: semantic UI, keyboard access, visible focus, labels, contrast, and assistive technology behavior are quality requirements.
- Internationalization: do not hardcode user-facing strings, dates, numbers, currencies, or layout assumptions when the app supports multiple locales.
- Privacy: collect the least data needed, protect PII, document retention/deletion/export behavior, and avoid leaking user data.
- Analytics: events should be purposeful, consistently named, owned, documented, and free of sensitive data.
- Background jobs: retries must be idempotent, failures visible, schedules documented, and queues observable.
- AI/LLM safety: prompts, tools, retrieval, data handling, evals, and fallbacks need explicit boundaries.
- Infrastructure: prefer declarative, reviewed, least-privilege infrastructure with safe defaults and environment parity.
- Documentation: docs should be accurate, concise, linked to decisions, and verified against code when used to guide agents.
- Multi-tenancy: every data read/write must be tenant-scoped when the product is tenant-aware.
- Payments: payment flows must be idempotent, auditable, reconciled, and safe around webhooks and retries.
- Notifications: user messaging needs templates, retries, unsubscribe/consent, delivery visibility, and privacy checks.
- Data import/export: parse defensively, validate before mutation, handle partial failure, and avoid leaking private data.
- API compatibility: public/client-facing contracts need versioning, deprecation, and backward-compatibility discipline.
- Developer experience: local setup, commands, seed data, and debugging paths should be documented and reproducible.
- Utilities: shared helpers should be small, named by domain purpose, tested when behavior is nontrivial, and not duplicate built-in or installed library features.
- Engineering: KISS before cleverness, DRY after the duplication is real, composition before inheritance, deletion before abstraction.
